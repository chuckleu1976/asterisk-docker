=== 3. Test SMS still has empty contact_id ===

SMS 155 arrived but showed empty contact_id.
Discovered: FILTER(SMS_FROM,0123456789+) was wrong!
FILTER(allowed_chars, string) — first arg is allowed chars, second is string.
FILTER(SMS_FROM,0123456789+) treats "SMS_FROM" (literal text) as allowed chars
and "0123456789+" as the string to filter!

Changed to:
- FILTER(0123456789+,${SMS_FROM})
- GotoIf fallback chain instead of IF

=== 5. SIM swap caused asterisk1 Rejected ===

User: "I have change the sim card in p1, and found looks like it effected
the ims registration status in p0, and asterisk1 show reject please found
out the reason."

Investigation findings:
- asterisk1 showed: ims.mncNone.mccNone.3gppnetwork.org (Rejected)
- asterisk2 showed: ims.mnc240.mcc310.3gppnetwork.org (Registered)
- File contents had mncNone.mccNone throughout pjsip.conf and epdg.conf

Root cause: read_sim() returns valid JSON dict even when IMSI read fails
(imsi=None, mcc=None, mnc=None). The update_epdg/pjsip functions don't
validate and write mncNone.mccNone. Once written, regex \d+ never matches
None, so re-running setup can never fix it.

Trigger: Sim swap triggered container restart via watch loop, loaded
already-corrupted config.

Fixes:
1. scripts/sim_docker.py: Added validation guards (skip update if missing)
2. Changed regex \d+ to \w+ to match None for recovery
3. Directly sed-fixed config/1/epdg.conf and config/1/asterisk/pjsip.conf
4. Force-recreated container

Confirmed asterisk1 Registered on ims.mnc240.mcc310.3gppnetwork.org.
Committed d794fcc.
