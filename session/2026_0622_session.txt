## Why startup is slow (~60–90s per container)

The delay is in entrypoint.sh, not from the SIM count:

| Step | Time | What it does |
|---|---|---|
| `sleep 3` | 3s | wait for pcscd socket |
| `ipsec start; sleep 2` | 2s | start strongSwan |
| Wait for `/tmp/pcscf_ip` | **up to 120s** (typically 10–30s) | strongSwan IKE_AUTH → ePDG → P-CSCF discovery |
| SIM reset | <1s | the new `RESET` step (cheap, 2 cards = 2 quick APDUs) |
| Asterisk core load | 15–30s | `asterisk -f` loads all modules, parses pjsip.conf, registers REGISTER |
| First REGISTER + 401 + RES + 200 OK | 1–3s | one AKA round-trip with the card |

So even with 2 SIMs you're paying **~30s for the IPsec tunnel and ~30s for Asterisk module load** = ~60s before REGISTER is even sent. The SIM-reset step is negligible (sub-second per card); it scales linearly only because `readers()` enumerates them.

The dominant cost is the IPsec/ePDG handshake. If you want to shave time:
- Lower the P-CSCF wait ceiling (`WAIT < 120`) — useless to wait that long; 60s is plenty.
- Increase IPsec re-initiate aggressiveness (already done with the backoff loop).
- Cannot speed up Asterisk module load without trimming `modules.conf`.

| Stage | Time |
|---|---|
| `sleep 3` + `ipsec start` + `sleep 2` | ~5s |
